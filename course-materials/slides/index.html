1.1 Sentry w workflow developera

Od "co właśnie odpaliło alert?" do decyzji o fixie w pięć minut — model obiektowy Sentry plus pełny przegląd cyklu życia Issue.

Object model — the only diagram you must memorize

Org → Team → Project → SDK → DSN → Event → Issue

  • Organization (devpowers) — SSO, billing, retention, rate limits.
  • Team — ownership routing and alert subscribers.
  • Project — one app or service. One DSN per platform. We have android, apple-ios, java-spring-boot, javascript-angular.
  • SDK + DSN — ingestion URL. Public by design; rate-limit and rotate if abused.
  • Event = one occurrence. Issue = grouped events with shared fingerprint. Release + environment are filters across both.

Sentry should answer five questions

Co · komu · od kiedy · w którym release · z jakim wpływem

  • What — exception, message, span anomaly, custom log.
  • Who — pseudonymous user ID, device, environment, geography.
  • When — first/last seen, frequency timeline, regression after release X.
  • Where in code — symbolicated stack, suspect commits, owners.
  • Impact — affected users, sessions, transactions, downstream services.

1.2 Projekty, SDK, custom context, prywatność

Zaprojektuj konfigurację tak, żeby pierwsze 100 eventów produkcyjnych przyszło użyteczne i bez wrażliwych danych — nie odwrotnie.

Punkty konfiguracji

DSN, environment, release, sample rate

  • DSN (Data Source Name) — publiczny URL, ustawiany przez env var; nie wbudowywać na stałe w mobilne binarki, których nie da się szybko rotować.
  • environment — dziś training; w realu production, staging, preview, development.
  • release — niezmienny identyfikator buildu, np. angular@<short-sha>. Ta sama wartość w SDK init i przy uploadzie source artefaktów.
  • tracesSampleRate — 1.0 w szkoleniu, w prod celuj w 0.05–0.20 ze względu na koszt.
  • sendDefaultPii — domyślnie false dla regulowanych branż. Włączamy field-by-field po data review.
Best practices per platforma — kliknij

Spring Boot: sentry.traces-sample-rate=0.05–0.20; rejestruj TracesSamplerCallback dla różnych route'ów. Release: -Dsentry.release=$RELEASE przy java -jar. Sampler nie powinien dotykać error eventów — te zawsze idą.

Angular: tracesSampleRate: 0.05 + replaysSessionSampleRate: 0.05 + replaysOnErrorSampleRate: 1.0 (zachowuje replay przy błędzie). Release: angular@$SHA identyczny w main.ts oraz sentry-cli sourcemaps upload.

Android: options.tracesSampleRate = 0.05. Release: nie ustawiaj ręcznie — SDK używa packageName@versionName+versionCode automatycznie. Override psuje join z Play Console.

iOS: tak samo — zostaw options.releaseName unset. dSYM upload wymaga, żeby ten identyfikator zgadzał się z tym, co poszło do App Store Connect.

Privacy baseline SURVEY

Scrubbing — w SDK i na poziomie projektu

  • Strona SDK: beforeSend / beforeBreadcrumb usuwa request body, tokeny, kody pocztowe, pola w kształcie PESEL/NIP/REGON.
  • Strona projektu: Settings → Security & Privacy → data scrubbers, sensitive fields, regex rules. To safety net, kiedy SDK code zregresuje.
  • Replay: maskAllText: true, blockAllMedia: true jako default — już wpięte w apps/angular/src/main.ts.
  • Android: options.isSendDefaultPii = false; nigdy nie loguj PESEL/postal; demo w SentryDemoApplication.kt.
  • "Nigdy nie osłabiamy scrubbingu, żeby demo było łatwiejsze" — jedyny twardy red line dzisiaj.
Whitelist vs. blacklist — dlaczego whitelisting wygrywa

Blacklist ("zmaskuj te pola PESEL/email/itd.") zawsze zostawia dziurę — kiedy programista doda nowe pole, ono wycieka.

Whitelist ("zmaskuj wszystko, unmask te konkretne klasy CSS / view tagi") jest auditowalny i bezpieczny by default.

Replay we wszystkich SDK Sentry implementuje whitelist mode jako domyślny. Angular: unmask: ['.safe-to-show']. Android: Modifier.sentryReplayUnmask(). iOS: .sentryReplayUnmask().

Custom context — co dodać, czego nigdy SURVEY

Tags vs. context vs. extra vs. user

  • Tags — indeksowalne, searchable, niska kardynalność (≤ ~100 unikalnych wartości na klucz). Dobre: coverage, vehicleType, riskBand. Złe: PESEL, request ID, quote.id.
  • Context — structured blobs, nie indeksowane. Dobre: parsed config, kształt failującego payloadu. Preferuj nad legacy extra.
  • User — pseudonimowy ID; nigdy email/telefon dla tej audience.
  • Breadcrumbs — automatyczne + manualna narracja. Ring buffer 100 wpisów — drop noise zanim zaleje sygnał.
  • Logs — structured key-value, skorelowane z eventem i trace. Sentry.logger.info(Sentry.logger.fmt`...`, {...}).
Anty-wzorce — czego unikać
  • Tag z message: "Failed to find policy 12345" → każdy ID nowa grupa → alert fatigue. Stable message + ID w context.
  • Wysoka kardynalność tag: quote.id jako tag → blow up tag aggregations + quota. Idzie do context.
  • Re-tagging device info na Androidzie/iOS: SDK już ustawia device.model, os.version, app.version. Duplikowanie = quota burn.
  • Sample 1.0 traces w prod: bilingowe span'y; spróbkuj 0.05–0.20, użyj tracesSampler dla per-route override.

The two install paths

Wizard now, manual when policy demands

  • Angular wizard: npx @sentry/wizard@latest -i sourcemaps --saas --org devpowers --project javascript-angular
  • Manual: npm i @sentry/angular + Sentry.init({...}) in main.ts + ErrorHandler/TraceService in app.config.ts. Exactly what is already in apps/angular/.
  • Spring wizard: not interactive; manual is the path. Add io.sentry:sentry-spring-boot-4-starter + io.sentry.jvm.gradle plugin. Already wired in apps/api-spring/build.gradle.kts.
  • DSN flow: build embeds DSN via env var; never commit it in mobile binaries; rotate by revoking the client key.

PolicyLab architecture

Angular → /api/quotes → Spring → Sentry

  • Angular sends POST /api/quotes with optional X-Demo-Scenario: normal|server-error|slow|validation|pii.
  • Spring services the request, emits a custom span around pricing rules, and tags the transaction with coverage / vehicleType / riskBand.
  • Browser tracing propagates a W3C trace header → Spring's OpenTelemetry layer joins it → one trace in Sentry.
  • For 500s the request is captured as a Sentry issue tied to that trace.
  • For validation 4xx no issue is raised — that's correct (expected client outcome), only logged.

1.4 Releases, source maps, R8/ProGuard, dSYM

Najbardziej setup-heavy moduł popołudnia, ale ten odblokowujący "czytelne stack trace w prod" — wprost powiązany z priorytetem z ankiety.

Release contract

Build once, upload once, deploy once

  • Pick SENTRY_RELEASE once — short commit SHA or immutable build ID. Same value across SDK init and source-artifact upload.
  • Order: choose release → build prod artifacts → create Sentry release → set commits → upload source maps/mappings/symbols → finalize → deploy → record deploy.
  • Source artifacts must be uploaded before events that need processing. Late uploads do not repair already-processed traces.
  • Never rebuild after upload — debug IDs would change and stack traces would silently mis-symbolicate.

Per-platform artifact map

Different artifact for each runtime

  • JavaScript / Angular — .js.map source maps with debug IDs. Generated by ng build --source-map. Upload via Sentry bundler plugin or sentry-cli sourcemaps inject/upload. Strip .map from deployable bundle.
  • Spring / JVM — JVM source bundle uploaded by the Sentry Gradle plugin (io.sentry.jvm.gradle). Gives source context inside stack frames.
  • Android — R8/ProGuard mapping SURVEY. The Sentry Android Gradle plugin uploads it per release. Without it the prod stack is unreadable.
  • NDK / native — debug symbols uploaded via sentry-cli debug-files upload.
  • iOS — .dSYM uploaded by the Xcode build phase the wizard adds.

Android R8/ProGuard — pełen walkthrough SURVEY

"Odzyskiwanie pełnych logów z Androida bez minify"

  • release build minifikuje i obfuskuje kod R8 (następca ProGuard) → klasy stają się a.b.c.
  • Sentry Android Gradle plugin (wpięty w apps/android/app/build.gradle.kts) generuje mapping.txt dla każdego buildu i uploaduje pod tag versionName+versionCode kiedy SENTRY_AUTH_TOKEN jest dostępny.
  • Sentry deobfuskuje po stronie serwera. Stack, który widzisz w Issue, jest oryginalnym Kotlin/Java.
  • Jeśli w prod nie widzisz prawdziwych nazw: build nie powinien zregenerować mappingu po uploadzie; versionName+versionCode w evencie musi się zgadzać z uploadem.
  • Weryfikacja: sentry-cli upload-dif --type=proguard mapping.txt dla ręcznego reuploadu z czystej maszyny.
Słownik: R8, ProGuard, dSYM, source map, source bundle
  • R8 — następca ProGuard, default minifier w Android Gradle Plugin od ~2019. Robi minify + obfuscate + tree-shaking w jednym przebiegu. Wynik: mapping.txt tłumaczący a.b.c → com.policylab.QuoteRepository.
  • ProGuard — starszy obfuscator; semantyka mapping.txt taka sama jak R8. Sentry obsługuje oba tym samym uploadem.
  • dSYM (Debug Symbols, Apple) — .dSYM bundle wygenerowany przez Xcode przy buildzie archive. Mapuje pamięciowe adresy native frames na symbole Swift/Obj-C. UUID dSYMu musi pasować do wydanej binarki.
  • source map — plik .js.map dla minifikowanego JavaScriptu. Mapuje minified frames na oryginalny TypeScript. Sentry używa Debug IDs (content-addressed) zamiast ścieżek, więc kolejność uploadu i deployu jest mniej krucha.
  • source bundle (JVM) — ZIP z plikami .java/.kt uploadowany przez Sentry Gradle plugin (includeSourceContext=true). Daje "source context" w stack trace (linie kodu wokół wyjątku) bez wystawiania źródeł publicznie.

CI/CD — dwa CLI, nie myl ich

sentry-cli (pipeline) vs. sentry (Ty)

  • sentry-cli — wąski, stabilny, do pipeline'ów. Auth = org token scoped org:ci. Używany w .github/workflows/release.yml, Bitbucket Pipelines, Jenkinsfile.
  • sentry (cli.sentry.dev) — szeroki, interaktywny, dla developerów i AI agents. Auth = user token przez sentry auth login. Tylko przy biurku; nie w buildzie.
  • CI token = source-map upload, release creation, code mappings. Nic więcej.
  • Dev/agent CLI = czyta wszystko, mutuje ostrożnie. Twoje uprawnienia user-poziomu.
🎯 Bonus: Sentry łapie też błędy CI/CD i deployu — bez tokenu

sentry-cli send-event działa tylko z DSN. DSN jest publiczny → idzie do vars, nie secrets. Każdy job ma teraz krok if: failure() który wysyła event do projektu odpowiadającego za to co się zepsuło.

npx --yes @sentry/cli@^2 send-event \
  --message "Spring release failed (spring-image) api-spring@a1b2c3d" \
  --level error \
  --environment release \
  --release "api-spring@a1b2c3d" \
  --tag "workflow:Release" \
  --tag "job:spring-image" \
  --tag "stage:release" \
  --tag "platform:spring" \
  --extra "run_url:https://github.com/.../actions/runs/123"

Trzy nowe repo variables (jednorazowo, już ustawione):

  • WEB_SENTRY_DSN — DSN projektu javascript-angular
  • API_SENTRY_DSN — DSN projektu java-spring-boot
  • ANDROID_SENTRY_DSN — DSN projektu android

Reusable action: .github/actions/sentry-report-failure/action.yml. Tagi workflow, job, stage, platform, sha, run_id + extra run_url. Filtruj w Sentry: environment:release stage:deploy.

Plus warstwa Spring: PolicyLabApplication.main() robi manual Sentry.init() przed SpringApplication.run() i łapie throwable boot-time. Crash z kontekstu Springa też idzie do Sentry — z tagiem boot.phase:pre-spring.

Tokens — który gdzie, dlaczego

One token type per scope, never mix

  • Project DSN — public, runtime env in each app. Rotate by revoking the client key.
  • Sentry org auth token (org:ci) — only in GitHub Actions secret. Build-time only. Created at devpowers.sentry.io/settings/auth-tokens.
  • Sentry user auth token — only on a developer's laptop via sentry auth login. Never in a workflow.
  • GitHub GITHUB_TOKEN — auto-injected; we use it to push images to ghcr.io (packages:write).
  • Coolify webhook URL — semi-secret; lives in the same GitHub training environment.
  • Full matrix & copy-paste commands: course-materials/guides/setup-tokens-and-deploy.md.

Co publiczne, co tajne

Kontrakt, który możesz pokazać security team

  • Publiczne: DSN (4 sztuki), publiczna FQDN deploya, nazwy obrazów GHCR, FQDN Coolify, kod źródłowy.
  • Tajne (GH env): SENTRY_AUTH_TOKEN, COOLIFY_WEBHOOK_URL.
  • Nigdy w obrazie, nigdy w compose: SENTRY_AUTH_TOKEN. Angular Dockerfile akceptuje go tylko jako build arg.
  • Nigdy na czacie / Teams / Slack: żaden token zaczynający się od sntrys_ ani URL z webhookiem Coolify.
  • Fork tego repo nie może odczytać sekretów ze środowiska training — taki jest sens environment-scoping.
Słownik skrótów na tym slajdzie — kliknij
  • FQDN (Fully Qualified Domain Name) — pełna nazwa domeny z subdomenami, np. policylab.edukey.ai. W przeciwieństwie do hostname'a (policylab), FQDN jest unikalny w globalnym DNS.
  • GHCR (GitHub Container Registry) — registry obrazów Docker zintegrowany z GitHub, host: ghcr.io. Każdy obraz należy do user/org GitHuba, np. ghcr.io/lucasmatuszewski/sentry-course-06-2026-web. Auth tym samym tokenem co GitHub Actions (GITHUB_TOKEN ma scope packages:write).
  • environment-scoping — w GitHub Actions sekret może być przypięty do "environment" zamiast do repo. Workflow musi explicite request ten environment (np. environment: training), wtedy dopiero ma dostęp do sekretu. Forki PR-ów nie mogą tego użyć.
  • build arg vs. runtime env — ARG w Dockerfile = wartość dostępna w czasie buildu, nie w runtime image. ENV = wartość w działającym kontenerze. SENTRY_AUTH_TOKEN używamy jako ARG (potrzebny do uploadu artifactów), nigdy jako ENV (kontener go nie potrzebuje).

1.5 Tracing, logi i Replay jako kontekst

Od "wyjątku" do "user kliknął tu, przeglądarka wysłała to, backend był wolny na tym spanie, DB zwróciła tamto" — jeden trace.

Tracing model

Transaction → spans → propagation SURVEY

  • Transaction to jedna user-facing operacja (load route, API call, message handler). Zawiera spans.
  • Trace propagation: nagłówki W3C traceparent + Sentry sentry-trace + baggage. Angularowy tracePropagationTargets ma /^\/api/ — dlatego Spring dołącza do śladu.
  • OpenTelemetry agent w Springu automatycznie tworzy spany HTTP. Custom spany na regułach pricingu mamy w SentryQuoteTelemetry.
  • Cały waterfall w jednym miejscu — nav we frontendzie, fetch, handler backendu, DB call.
  • Sampling: tracesSampleRate=1.0 dziś, w prod celuj 0.05–0.20. Eventy z errorem dostają transaction by default.
Best practices: tracing w produkcji
  • Spring: tracesSampler per route — POST /api/quotes 50%, healthcheck 0%. Wymaga zbioru danych zanim zaczniesz tuning.
  • Angular: tracePropagationTargets tylko first-party hosts. Nigdy nie propaguj do third-party CDN, analytics, ad networks — ich serwery odrzucają nieznane nagłówki albo (gorzej) je logują.
  • Android/iOS: dodaj SentryOkHttpInterceptor / URLSession integration. Mobilne sesje są długie; sample 0.05.
  • Spans są bilingowe; error eventy nie. Sampler nigdy nie powinien blokować error eventów.
  • Custom span tylko tam gdzie chcesz widzieć latencję jako label (rule evaluation, DB query batch). Mniej znaczy więcej.

Structured logs SURVEY

"Dołączanie customowych pól do logów"

  • Sentry Logs is the structured-log surface inside Sentry, correlated to issues and traces.
  • Angular: Sentry.logger.info(Sentry.logger.fmt`User ${id} clicked`, { action: 'click' }).
  • Spring: sentry.logs.enabled=true already set; LOGGER.info("Quote created: coverage={}, riskBand={}", c, r) is what you saw in QuoteController.
  • Android: same Logback-style API via SentryAndroid.
  • Custom fields ride as structured key-value pairs — searchable in the UI, filterable by tag.
  • Privacy reminder: structured ≠ free pass to log PII. Same scrubbing rules apply.

Session Replay

What the user actually saw, with masking

  • 10% of sessions sampled, 100% on error — already in main.ts.
  • maskAllText: true + blockAllMedia: true by default. Unmask explicitly with CSS class — never the other way around.
  • Replay link appears in the Issue right rail. One click → DOM playback aligned to the moment of error.
  • Cost model: per-session storage. Sample low in prod; rely on the on-error trigger.

1.6 Monitory, alerty i redukcja szumu

Dopasuj alerty do tego jak zespół naprawdę reaguje na przerwy — Teams, Jira, on-call — i unikaj najczęstszej pułapki alert fatigue.

Typy alertów w Sentry

Issue, metric, anomaly, uptime, cron

  • Issue alerts — first-seen, regression, spike częstotliwości per environment.
  • Metric alerts — error rate, p95 transaction duration, span throughput.
  • Anomaly — automatyczny baseline; przydatne dla noisy services gdzie thresholdy zmieniają się w czasie.
  • Uptime monitor — syntetyczny ping na publiczny endpoint.
  • Cron — heartbeat ze scheduled joba; alarmuje kiedy job przestaje się meldować.

Jira + Bitbucket integration JIRA WORKFLOW

Two-way link, nie copy-paste workflow

  • Issue alert rule → akcja "Create Jira ticket". Sentry stempluje issue Jira key; Jira ticket linkuje z powrotem do Sentry.
  • Native Bitbucket integration (Settings → Integrations → Bitbucket) odblokowuje suspect commits, code mappings, "Create Bitbucket issue" action — to jest the unlock dla insurance teamu na Bitbucket Cloud/Server.
  • Ownership rules: regex/path-based mapping kod → team → Jira board. docs.sentry.io/product/issues/ownership-rules.
  • Dla regulowanego stacka: Jira jest system of record; Sentry karmi go danymi. Nie próbuj robić workflow w Sentry comments.
  • W organizacji szkoleniowej nie włączamy integracji live — opisujemy okablowanie.
Krok po kroku: jak podłączyć Sentry ↔ Bitbucket ↔ Jira
  1. Bitbucket integration: Sentry Settings → Integrations → Bitbucket → Install. Autoryzuj OAuth dla workspace'u. Wybierz repozytoria do podpięcia.
  2. Code mappings: Sentry sam zaproponuje mapping ścieżek (np. com/policylab/api/quote/QuoteService.java → repo path apps/api-spring/src/main/java/.../QuoteService.java). Akceptuj.
  3. Suspect commits działają automatycznie kiedy: (a) release ma set-commits (robi to getsentry/action-release); (b) code mappings są skonfigurowane; (c) Bitbucket integration jest podpięty.
  4. Jira integration: Sentry Settings → Integrations → Jira Cloud/Server. Authorize, wybierz projekty Jira. Wynik: w każdym Issue Sentry pojawia się "Create Jira Ticket" + "Link to Existing".
  5. Alert rule action: Alerts → Create alert → "Then perform these actions" → "Create a Jira ticket in [project] of type [Bug]". Konfiguruje się raz, automatyzuje workflow.
  6. Bonus: kiedy Sentry release zawiera commit naprawiający bug, Sentry oznacza Issue jako resolved-in-release automatycznie. Trend wykresu robi się sam.

Demo na żywo wymaga admin w organizacji Sentry (nie dla uczestników kursu). Jeśli mamy czas w długiej przerwie — możemy razem podłączyć Bitbucket sandbox.

Noise reduction playbook

Jak nie wysłać 3:00 AM page o niczym

  • Zawsze scope alertów do environment:production. Nigdy nie pozwól błędowi ze staging obudzić on-call.
  • Threshold: new issue w ostatniej 1h z >5 affected users bije jakikolwiek error się dzieje.
  • Throttle: per-rule cooldown żeby uniknąć alert storms podczas incydentu.
  • Ownership routing: paguj team który posiada plik, nie "kanał".
  • Inbound filters i ignoreErrors dla znanych noisy patterns (browser extensions, dead bots).
  • Kwartalny review alert rules — jeśli alert nigdy nie odpalił albo zawsze jest dismissed, kill go.
CI/CD przykłady w tym repo — co jest gotowe, co nie
  • GitHub Actions (.github/workflows/) — pełen, działający, produkcyjny przykład. ci.yml uruchamia się na każdym PR; release.yml buduje obrazy, uploaduje source artefacts, deployuje na Coolify.
  • Bitbucket Pipelines (course-materials/ci-cd/bitbucket-pipelines.example.yml) — annotated starting point. Wymaga ~30 min dostosowania: dodać branch trigger, dopasować ścieżkę dist, dopiąć deployment environment, dodać Java/Spring job (obecnie tylko Angular).
  • Jenkinsfile (course-materials/ci-cd/Jenkinsfile.example) — starting point. Wymaga ~30 min: prawdziwy agent label, tools { jdk } block dla Springa, SCM stage dla Bitbucket Server.
  • release-contract.md — provider-neutral checklist; demo-ready as-is.

Honest framing dla audience: "Pokazuję GitHub na żywo, dla Bitbucket i Jenkins dostajecie wzorce do replikacji. Native Bitbucket integration w Sentry to osobny temat — odblokowuje suspect commits, niezależnie od pipeline'u."

Device filtering SURVEY

"Filtrowanie po konkretnych urządzeniach"

  • Sentry automatycznie taguje każdy event: device.model, device.manufacturer, os.name, os.version, screen, locale, battery, network type.
  • Filtr w Issue list: device.model:"SM-S928B" albo os.name:Android os.version:>=15.
  • Cross-issue analiza w Explore (Discover): aggregate by device.model → zobacz które urządzenia dominują w konkretnym wyjątku.
  • Jeśli wysyłasz per-OEM hotfix, alert scoped device.manufacturer:samsung może go uruchomić.
Best practice: nie duplikuj auto-tagów

Sentry Android SDK już ustawia device.model, device.manufacturer, os.name, os.version, app.version. Ręczny options.setTag("device_model", Build.MODEL) = duplikacja + quota burn + risk różnicy w wartości.

Twoje tagi powinny dodawać wymiary których SDK nie zna: policyType, userTier, experimentVariant. Kardynalność ≤ ~100 unikalnych wartości na klucz.

Custom fields on Android SURVEY

"Customowe pola w logach"

  • Already wired in SentryDemoApplication.kt: global tags course-id, developer-level, target-organization, host-locale.
  • Per-event scope: Sentry.configureScope { it.setTag("policy.id", id); it.setContext("vehicle", mapOf(...)) }.
  • Per-thread configureScope in coroutines — show how scopes propagate across withContext.
  • Logs API: Sentry.logger().info("Quote requested", mapOf("scenario" to scenario)).

1.7 AI w Sentry — Seer, MCP, dev CLI

Krótki moduł by design. Celem jest pokazać AI jako dodatek, nie główny workflow — ankieta dała Sentry-AI interest ~3.3/5.

Seer

Treat root-cause AI as a hypothesis

  • Seer reads the issue, the linked code, and recent events; proposes a root cause and sometimes a patch.
  • Workflow: write your one-line hypothesis first, then ask Seer to explain evidence and uncertainty.
  • Verify cited frames, spans, and release against the actual event before trusting.
  • Never accept a generated patch that broadens data collection or hides exceptions.
  • Right rail on any Issue — same place every time.

Sentry MCP + dev CLI

Your IDE/agent talks to Sentry; you still drive

  • Sentry MCP (mcp.sentry.dev) — Claude / Cursor / JetBrains Junie / Zed can query issues, traces, releases through a tool interface.
  • Auth = least-privilege, short-lived org-owned token; scope to specific projects.
  • Good prompt: "In java-spring-boot staging, summarize evidence for issue PL-API-12. Separate facts from hypotheses. Exclude request body."
  • Bad prompt: "Fix production."
  • sentry dev CLI: sentry issue explain <id>, sentry issue plan <id> are MCP-backed AI features at the terminal.
  • Reference: course-materials/guides/09-seer-mcp.md.
