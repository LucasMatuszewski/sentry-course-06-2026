1.1 Sentry w workflow developera

From "what alert just fired?" to a fix decision in five minutes — covering the Sentry object model and a full issue lifecycle walk-through.

Object model — the only diagram you must memorize

Org → Team → Project → SDK → DSN → Event → Issue

  • Organization (devpowers) — SSO, billing, retention, rate limits.
  • Team — ownership routing and alert subscribers.
  • Project — one app or service. One DSN per platform. We have android, apple-ios, java-spring-boot, javascript-angular.
  • SDK + DSN — ingestion URL. Public by design; rate-limit and rotate if abused.
  • Event = one occurrence. Issue = grouped events with shared fingerprint. Release + environment are filters across both.

Sentry should answer five questions

Co · komu · od kiedy · w którym release · z jakim wpływem

  • What — exception, message, span anomaly, custom log.
  • Who — pseudonymous user ID, device, environment, geography.
  • When — first/last seen, frequency timeline, regression after release X.
  • Where in code — symbolicated stack, suspect commits, owners.
  • Impact — affected users, sessions, transactions, downstream services.

1.2 Projekty, SDK, custom context, prywatność

Design the configuration so the first 100 production events arrive useful and non-sensitive — not the other way around.

Configuration anchors

DSN, environment, release, sample rate

  • DSN — runtime, env var, never hard-coded in mobile binaries you cannot rotate quickly.
  • environment — training today; in real life production, staging, preview, development.
  • release — immutable build identifier, e.g. angular@<short-sha>. Same value embedded at build time and used to upload source artifacts.
  • tracesSampleRate — 1.0 for training, target 0.05–0.20 in prod for cost.
  • sendDefaultPii — default false for regulated stacks. Opt in only after a data review.

Privacy baseline SURVEY

Data scrubbing — at SDK and at project level

  • SDK side: beforeSend / beforeBreadcrumb to strip request bodies, tokens, postal codes, PESEL-shaped fields.
  • Project side: Settings → Security & Privacy → data scrubbers, sensitive fields, regex rules.
  • Replay: maskAllText: true, blockAllMedia: true are our defaults — already wired in apps/angular/src/main.ts.
  • Android: options.isSendDefaultPii = false; never log PESEL/postal code; demo this in SentryDemoApplication.kt.
  • "Never weaken scrubbing to make a demo easier" — this is the only hill we die on today.

Custom context — what to add, what to never add SURVEY

Tags vs. extra vs. user vs. context

  • Tags — indexable, searchable, low-cardinality. Good: coverage, vehicleType, riskBand, app.locale. Bad: PESEL, request ID.
  • Extra / context — high-cardinality blobs, not indexed. Good: parsed config, structured shape of the failing payload.
  • User — pseudonymous ID, never email/phone for this audience.
  • Breadcrumbs — automatic + manual narrative.
  • Logs — structured key-value, correlated to event and trace. Sentry.logger.info(Sentry.logger.fmt`...`, {...}).

Angular + Spring Boot 4 — onboarding na żywo

Wizard vs manual install, DSN flow, verify event, and trace propagation from the browser through /api/quotes.

The two install paths

Wizard now, manual when policy demands

  • Angular wizard: npx @sentry/wizard@latest -i sourcemaps --saas --org devpowers --project javascript-angular
  • Manual: npm i @sentry/angular + Sentry.init({...}) in main.ts + ErrorHandler/TraceService in app.config.ts. Exactly what is already in apps/angular/.
  • Spring wizard: not interactive; manual is the path. Add io.sentry:sentry-spring-boot-4-starter + io.sentry.jvm.gradle plugin. Already wired in apps/api-spring/build.gradle.kts.
  • DSN flow: build embeds DSN via env var; never commit it in mobile binaries; rotate by revoking the client key.

PolicyLab architecture

Angular → /api/quotes → Spring → Sentry

  • Angular sends POST /api/quotes with optional X-Demo-Scenario: normal|server-error|slow|validation|pii.
  • Spring services the request, emits a custom span around pricing rules, and tags the transaction with coverage / vehicleType / riskBand.
  • Browser tracing propagates a W3C trace header → Spring's OpenTelemetry layer joins it → one trace in Sentry.
  • For 500s the request is captured as a Sentry issue tied to that trace.
  • For validation 4xx no issue is raised — that's correct (expected client outcome), only logged.

1.4 Releases, source maps, R8/ProGuard, dSYM

The most expensive afternoon module in setup terms, but the one that unlocks "readable stack traces in prod" — directly tied to a survey priority.

Release contract

Build once, upload once, deploy once

  • Pick SENTRY_RELEASE once — short commit SHA or immutable build ID. Same value across SDK init and source-artifact upload.
  • Order: choose release → build prod artifacts → create Sentry release → set commits → upload source maps/mappings/symbols → finalize → deploy → record deploy.
  • Source artifacts must be uploaded before events that need processing. Late uploads do not repair already-processed traces.
  • Never rebuild after upload — debug IDs would change and stack traces would silently mis-symbolicate.

Per-platform artifact map

Different artifact for each runtime

  • JavaScript / Angular — .js.map source maps with debug IDs. Generated by ng build --source-map. Upload via Sentry bundler plugin or sentry-cli sourcemaps inject/upload. Strip .map from deployable bundle.
  • Spring / JVM — JVM source bundle uploaded by the Sentry Gradle plugin (io.sentry.jvm.gradle). Gives source context inside stack frames.
  • Android — R8/ProGuard mapping SURVEY. The Sentry Android Gradle plugin uploads it per release. Without it the prod stack is unreadable.
  • NDK / native — debug symbols uploaded via sentry-cli debug-files upload.
  • iOS — .dSYM uploaded by the Xcode build phase the wizard adds.

Android R8/ProGuard — explicit walkthrough SURVEY

"Odzyskiwanie pełnych logów z Android bez minify"

  • release build minifies and obfuscates with R8 → stack frames become a.b.c.
  • The Sentry Android Gradle plugin (already in apps/android/app/build.gradle.kts) generates a mapping file per build and uploads it tagged by versionName+versionCode when SENTRY_AUTH_TOKEN is exported.
  • Sentry de-obfuscates on the server. The stack you see in the Issue is the original Kotlin/Java.
  • If you don't see real names in prod: check that the build did not regenerate the mapping after upload, and that the versionName+versionCode in the event matches the uploaded mapping.
  • Verification: sentry proguard upload for a manual reupload from a clean machine.

CI/CD — two CLIs, do not confuse

sentry-cli (pipeline) vs. sentry (you)

  • sentry-cli — narrow, stable, designed for pipelines. Auth = org token scoped org:ci. Used in .github/workflows/release.yml, Bitbucket Pipelines, Jenkinsfile.
  • sentry (cli.sentry.dev) — broad, interactive, for developers and AI agents. Auth = your user via sentry auth login. Use it at your desk; do not bake it into builds.
  • CI token = source-map upload, release creation, code mappings. That's it.
  • Dev/agent CLI = read everything, mutate carefully. Your user permissions apply.
  • Reference: course-materials/guides/09-seer-mcp.md § "Two Sentry CLIs".

1.5 Tracing, logi i Replay jako kontekst

From "an exception" to "the user clicked here, the browser sent that, the backend was slow on this span, the DB returned that" — one trace.

Tracing model

Transaction → spans → propagation SURVEY

  • A transaction is one user-facing operation (route load, API call, message handler). Contains spans.
  • Trace propagation: W3C traceparent + Sentry sentry-trace + baggage headers. Angular's tracePropagationTargets includes /^\/api/ — that is why Spring joins the trace.
  • Spring's OpenTelemetry agent emits HTTP spans automatically. Custom spans on pricing rules already exist in SentryQuoteTelemetry.
  • You see the whole waterfall in one place — frontend nav, fetch, backend handler, DB call.
  • Sampling: tracesSampleRate=1.0 today, target 0.05–0.20 in prod. Errors get a transaction by default.

Structured logs SURVEY

"Dołączanie customowych pól do logów"

  • Sentry Logs is the structured-log surface inside Sentry, correlated to issues and traces.
  • Angular: Sentry.logger.info(Sentry.logger.fmt`User ${id} clicked`, { action: 'click' }).
  • Spring: sentry.logs.enabled=true already set; LOGGER.info("Quote created: coverage={}, riskBand={}", c, r) is what you saw in QuoteController.
  • Android: same Logback-style API via SentryAndroid.
  • Custom fields ride as structured key-value pairs — searchable in the UI, filterable by tag.
  • Privacy reminder: structured ≠ free pass to log PII. Same scrubbing rules apply.

Session Replay

What the user actually saw, with masking

  • 10% of sessions sampled, 100% on error — already in main.ts.
  • maskAllText: true + blockAllMedia: true by default. Unmask explicitly with CSS class — never the other way around.
  • Replay link appears in the Issue right rail. One click → DOM playback aligned to the moment of error.
  • Cost model: per-session storage. Sample low in prod; rely on the on-error trigger.

1.6 Monitory, alerty i redukcja szumu

Match alerts to the way the team actually receives interruption — Teams, Jira, on-call — and avoid the most common alert-fatigue trap.

Alert types in Sentry

Issue, metric, anomaly, uptime, cron

  • Issue alerts — first-seen, regression, frequency spike per environment.
  • Metric alerts — error rate, p95 transaction duration, span throughput.
  • Anomaly — automatic baseline; useful for noisy services where thresholds drift.
  • Uptime monitor — synthetic ping on a public endpoint.
  • Cron — heartbeat from a scheduled job; pages you when it stops checking in.

Jira + Bitbucket integration JIRA WORKFLOW

Two-way link, not a copy-paste workflow

  • Issue alert rule → "Create Jira ticket" action. Sentry stamps the issue with the Jira key; Jira ticket links back to the Sentry issue.
  • Bitbucket integration provides commit linking + suspect-commit detection on PRs.
  • Ownership rules: regex/path-based mapping of code → team → Jira board.
  • For a regulated stack: keep Jira as the system of record; Sentry feeds it. Don't try to use Sentry comments as workflow.
  • For the workshop org we do not enable Jira/Bitbucket integrations live — we describe the wiring instead.

Noise reduction playbook

How to not send a 3 AM page for nothing

  • Always scope alerts to environment:production. Never let a staging error wake on-call.
  • Threshold: new issue in last 1h with >5 users affected beats any error happens.
  • Throttle: per-rule cooldown to avoid alert storms during incidents.
  • Ownership routing: page the team that owns the file, not "the channel".
  • Inbound filters and ignoreErrors for known-noisy patterns (browser extensions, dead bots).
  • Quarterly alert-rule review — if an alert never fires or is always dismissed, kill it.

1.7 AI w Sentry — Seer, MCP, dev CLI

Short module by design. The point is to make AI a supplement, not the primary workflow — survey put Sentry-AI interest at ~3.3/5.

Seer

Treat root-cause AI as a hypothesis

  • Seer reads the issue, the linked code, and recent events; proposes a root cause and sometimes a patch.
  • Workflow: write your one-line hypothesis first, then ask Seer to explain evidence and uncertainty.
  • Verify cited frames, spans, and release against the actual event before trusting.
  • Never accept a generated patch that broadens data collection or hides exceptions.
  • Right rail on any Issue — same place every time.

Sentry MCP + dev CLI

Your IDE/agent talks to Sentry; you still drive

  • Sentry MCP (mcp.sentry.dev) — Claude / Cursor / JetBrains Junie / Zed can query issues, traces, releases through a tool interface.
  • Auth = least-privilege, short-lived org-owned token; scope to specific projects.
  • Good prompt: "In java-spring-boot staging, summarize evidence for issue PL-API-12. Separate facts from hypotheses. Exclude request body."
  • Bad prompt: "Fix production."
  • sentry dev CLI: sentry issue explain <id>, sentry issue plan <id> are MCP-backed AI features at the terminal.
  • Reference: course-materials/guides/09-seer-mcp.md.
