Fundamentals

The Sentry object model

  • Organization → SSO, billing, retention
  • Team → ownership, alert routing
  • Project → one app or service, one DSN per platform
  • SDK + DSN → ingestion endpoint, public by design
  • Event → one occurrence; Issue → grouped events

Issue lifecycle

Capture → group → triage → resolve

  • Capture: SDK normalises and tags the event
  • Group: fingerprint based on stack + message
  • Assign: ownership rules or manual
  • Triage: priority, environment, release filters
  • Resolve: by commit, release, or "next release"
  • Regression: auto-reopen on a later release
10:10–11:00

Configuration

Projects & SDK options that matter

  • environment: prod / staging / dev — never lump together
  • release: tie events to a build artifact
  • tracesSampleRate / profilesSampleRate: cost lever
  • inAppInclude: your code vs vendor noise
  • beforeSend: last-mile scrubbing & routing

Privacy baseline

Default to less data

  • send-default-pii=false — opt in, never opt out
  • Pseudonymous user IDs, never raw email / login
  • Narrow custom context — only what helps triage
  • Server-side scrubbing + client-side beforeSend
  • Document what you collect — review with security & legal

13:30–14:30

Releases

Source maps, source context, symbolication

  • One release per build; tie to a git SHA
  • Web: upload JS source maps via sentry-cli releases
  • Android: R8/ProGuard mapping uploaded per build variant
  • iOS: dSYMs uploaded from CI, not from a dev laptop
  • Without symbols you have stack traces of nothing

CI / CD

Release job, scoped token

  • GitHub Actions job: create release, associate commits, upload artifacts, finalize
  • SENTRY_AUTH_TOKEN stored as a repo / org secret
  • Token scope: org:ci + project release write
  • Rotate quarterly; never echo, never commit
  • Non-live Bitbucket / Jenkins examples in course-materials/ci-cd/
14:30–15:30

Performance

Tracing, logs, Replay, profiling

  • Spans show what your code did; traces stitch them across services
  • Structured logs attached to the trace — searchable, correlated
  • Replay surfaces what the user did before the error
  • Profiling: function-level hotspots, no separate agent
  • Sampling is a cost lever — model it, don't guess it
